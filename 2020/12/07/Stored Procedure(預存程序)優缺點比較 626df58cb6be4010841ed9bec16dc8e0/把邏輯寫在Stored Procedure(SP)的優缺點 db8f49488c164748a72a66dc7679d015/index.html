把邏輯寫在Stored Procedure(SP)的優缺點

先說結論：我認為缺點大於優點

雖然寫在SP內也是一種快速建立解決方案的做法，就要看整個架構是否願意支持這樣的做法。

缺點

  1. 邏輯容易四散各地

    我認為這是反對寫在SP內最大的理由。

    SP裡面有許多判斷邏輯，但程式裡面可能到處也都有判斷邏輯，這樣會造成混淆。且系統持續刊發或後續進入維運增修階段，很難管理developer不小心將邏輯寫在應用程式內。

    久而久之沒人維護(follow)這個架構，就會愈來愈亂。

    舉一個實際遇到的例子：
    有個功能要是[新增廠商資料]，但會判斷”是否重複新增”，這個判斷寫在SP裡面，如果有重複新增資料，就會回傳新增失敗。
    而”驗證資料是否為空值”的相關判斷，卻是寫在應用系統內，SP本身不會驗證空值，所以如果從其他地方呼叫SP進來一堆空值是OK的。
    在另一個系統功能的地方，也是可以增加廠商資料，但卻沒有擋空值，呼叫了同一個SP，導致裡面的資料有些是空值有些又不是
    這樣整個邏輯未來要追查就會變得不知道要找SP還是找應用系統，因為兩邊對於這個資料表都有邏輯寫在裡面。

  2. 程式撰寫不易

    SQL本身不是設計用來做複雜判斷的，所以if else、try catch等都是滿後面才加入的語法功能，但撰寫上又沒有IDE良好的支援，撰寫跟debug上都更困難。

    且在開發時，除了寫應用程式，還要進到DB找SP的邏輯，同時處理兩邊對於developer不友善，效率也沒有較高，不如全部寫在應用程式中。

  3. 應用系統沒辦法用IDE直接除錯

    這點最麻煩，SP執行雖然也是有中斷點，但等於是要在原本coding IDE之外，再特別來處理這塊。

  4. 不符合關注點分離

    前後端都愈拆愈開了，DB跟後端也是相同的道理。讓應用系統專注於邏輯處理，資料庫的專注於取出對應的資料，後續的操作交由應用程式用ORM處理。

優點

  1. 降低連線次數

    呼叫SP一次做大量的事情，可以減少連線次數

  2. 跨資料庫存取快速

    同第一點，因為SP可以直接存取不同的資料庫，在一次連線內就可以完成。

    如果寫在程式內，可能要各自取出不同DB來源的資料，再從程式內組合起來。

  3. 權限控管

    簡單說概念就是等於用SP做一個封裝！

    如果對於資料庫進行控管，就是可能某一個角色可以使用特定的資料庫

    但管理的範圍比較大，不能設定某些帳號不能讀取某些table(這部分我不確定，要再確認)

    如果寫在SP裡面，就可以針對特定SP的使用權限控管，SP可以增刪改查某個table，

  4. 如果有多個client應用系統，都需要使用的邏輯，放在SP就不用每套應用系統都自己寫一次邏輯。
    (其實也可以寫成共用底層元件或現在最流行的API，提供給各個應用系統使用，但就需要有版本跟管理的議題。而放在SP的話，限制則是變成邏輯只有唯一一個最新版本，無法針對client端支援不同版本的邏輯，這就看系統架構是需要哪一種了。)
